## 1. Purpose

When `blueprint generate --namespace acme` runs, the generated code uses
`Acme_PaymentsContract` and `Acme_StripeAdapter`. An attacker who sees
`Acme_PaymentsContract` in minified JS can still infer the system's
architecture because the pattern `*_PaymentsContract` is recognisable across
all Blueprint users.

Aliasing replaces the contract names entirely with project-specific names.
Instead of `initiatePayment` the generated function is called `chargeCustomer`.
The original contract name does not appear anywhere in the generated output.

### Security Dependency

The alias system provides its security value only when contracts are excluded
from the npm package. The npm package (`@friehub/blueprint`) ships a stripped
catalog (`catalog.min.json`) that contains no function signatures, no
invariants, no temporal values, and no error taxonomies. Raw contracts are
not included in the package.

If raw contracts were present in the installed package, an attacker could
install the package and trivially map aliased names back to contract names
by reading the original function signatures alongside the generated code.
The alias system would provide zero security value in that scenario.

**The alias system is a second layer of protection.** The first layer is
ensuring contracts do not ship in the npm package. Both layers must be in
place for the alias system to provide meaningful security.

## 2. File format

Aliases live in a separate file `blueprint.aliases.json5` at the project root.
Uses JSON5 so comments are supported. This file is never committed to version
control -- generated by a project-specific seed script and stored in CI secrets
or a key management system.

```json5
{
  // Version for format compatibility
  version: "1.0",

  // Function name aliases
  functions: {
    initiatePayment: "chargeCustomer",
    verifyPayment:  "confirmCharge",
    refundPayment:  "reverseTransaction",
  },

  // Module name aliases (not yet implemented)
  modules: {
    payments: "billing",
    auth:     "identity",
    users:    "accounts",
  },

  // Class name aliases (not yet implemented)
  classes: {
    StripeAdapter: "CardProcessor",
    RedisAdapter:  "CacheStore",
    ClerkAdapter:  "IdentityProvider",
  },

  // Config field aliases (not yet implemented)
  config: {
    api_key:        "stripe_secret",
    webhook_secret: "stripe_webhook_token",
  },
}
```

## 3. Aliasing dimensions

### 3.1 Function aliases (`functions`) -- IMPLEMENTED

Maps contract function names to generated function names. Applied in all 5
language generators (TypeScript, Python, Go, Rust, Java).

Usage:

```bash
blueprint generate --lang typescript --aliases blueprint.aliases.json5
```

- Interfaces, adapter stubs, and conformance tests all use the aliased names.
- `--namespace` is applied after aliasing, so `getData` aliased to `fetchData`
  with `--namespace acme` becomes `acme_fetchData`.
- Unaliased functions keep their contract names.

### 3.2 Module aliases (`modules`) -- NOT IMPLEMENTED (B-4)

Maps contract module names to generated directory and file names.

- Changes generated file paths: `interfaces/payments.ts` -> `interfaces/billing.ts`
- Changes generated directory structure: `adapters/payments/stripe.ts` ->
  `adapters/billing/stripe.ts`
- Changes import statements across all generated files

### 3.3 Class aliases (`classes`) -- NOT IMPLEMENTED (B-4)

Maps adapter class names to project-specific class names.

- Overrides the default `{Provider}Adapter` naming pattern
- Applies to class declarations, constructor invocations, test instantiations
- Does NOT affect the module name in file paths (use module aliases for that)

### 3.4 Config field aliases (`config`) -- NOT IMPLEMENTED (B-5)

Maps adapter constructor parameter names to project-specific names.

- Changes constructor parameter names in generated adapter classes
- Changes environment variable names in scoped `.env.example` output
- Uses aliased names in env vars (security, not contract names)
- Does NOT affect the internal SDK call logic

## 4. Loading and merging -- IMPLEMENTED

1. Pass `--aliases path/to/blueprint.aliases.json5` to `blueprint generate`.
2. The file uses JSON5 format (comments supported, no quotes on keys needed).
3. Aliases are loaded and passed through `GeneratorContext` to all generators.
4. When both `--namespace` and aliases are present, namespace is applied AFTER
   aliasing. Example: module `payments` aliased to `billing` with
   `--namespace acme` produces `Acme/interfaces/billing.ts` and
   `Acme_BillingContract`.
5. If the file does not exist, no aliasing is applied (silent fallback).

---

## 5. Partial aliases -- WORKS BY OMISSION

Not every function needs an alias. Unaliased functions keep their contract
names. An empty object in any dimension means "no aliases applied for this
dimension":

```json5
{
  version: "1.0",
  functions: {},             // no function aliasing
  modules: { payments: "billing" },  // module alias (not yet effective)
  classes: {},
  config: {},
}
```

## 6. Error handling

| Condition | Behavior |
|---|---|
| Alias target matches a different contract function name | Warning emitted, alias applied |
| Alias source does not exist in current catalog version | Warning emitted, alias ignored |
| Two aliases produce the same target name | Error emitted, generation aborted |
| File is malformed JSON/JSON5 | Error emitted, generation aborted |
| Unknown dimension keys | Ignored with a warning |

## 7. verify command integration -- NOT IMPLEMENTED (B-8)

`blueprint verify` must reverse the alias map before checking compliance:

1. Accept `--aliases` flag (same file used during generation)
2. Build a reverse map: `{aliased_name: contract_name}`
3. For each function in the implementation, reverse-map to contract name
4. Check compliance against the contract using reverse-mapped names

Namespace prefix must be stripped before reverse-mapping.

When verification fails, error messages show both names:

```
Missing function: "signIn" (aliased as "authenticate")
```

---

## 8. Out of scope for v1

- Per-environment alias files (dev vs prod aliases)
- Automated alias generation from existing codebase
- Alias rotation without regeneration
- CI enforcement for high-value module coverage

---

## 9. Resolved decisions

All settled. No remaining open questions.

1. **JSON5** -- `.json5` extension, supports comments.
2. **Regex aliasing** -- Out of scope for v1.
3. **`--dump-aliases`** -- Verify command flag (not yet implemented).
4. **Config aliases in `.env.example`** -- Uses aliased names.
5. **Language convention** -- User provides aliases in the target language's
   convention (snake_case for Python/Rust, PascalCase for Go, camelCase for
   TS/Java). No automatic case conversion.
